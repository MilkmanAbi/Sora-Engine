# PaperDesign
### A design language for tools that look like they were drawn, not inflated.

Version 0.1 — Personal specification

---

## 0. Why this exists

Every major design system optimizes for something. Material optimizes for "bold and graphic" — which in practice means everything gets bigger, rounder, and louder until the interface reads like a toy. Apple's HIG optimizes for restraint and depth. Swiss/International Typographic Style (Müller-Brockmann, the whole "grid systems" lineage) optimizes for order — a page is a set of columns and units, and nothing sits outside that structure by accident. Dieter Rams' Braun-era principles optimize for honesty: a control looks like what it does, nothing is decorated to seem more important than it is. Microsoft's Fluent 2 optimizes for *adaptability* — one token system that reshapes itself per platform while staying recognizably one product family, and a small, deliberate shape vocabulary (rectangle, circle, pill, and a "beak" for things that point at their anchor) rather than an open-ended one. Firefox's Acorn system optimizes for *trust through separation* — a hard visual line between "chrome" (the browser, which you trust) and "content" (the page, which you don't, by default) — plus just enough restrained personality in copy and motion to feel made by people, not generated by a template.

PaperDesign borrows from all six and rejects one habit several of them share: **using visual weight as a substitute for hierarchy.** Big rounded corners, oversized touch targets, generous whitespace — these are shortcuts that make an interface *feel* friendly without making it *clear*. PaperDesign gets clarity from structure (the grid) and restraint (the palette, the shape language), not from inflating everything.

The name is literal. Think of a well-kept engineering notebook or a sheet of graph paper — every element sits on a rule, corners are square because paper corners are square, and the only softness comes from the writing on top of it, not from the page itself.

This version adds one more thing on top of the structure: **a system can be perfectly gridded and still feel dead if nothing about it responds like a physical object would.** That's the specific thing Fluent's "Light" and "natural motion" guidance gets right, and it's the specific thing Anthropic's brand work gets right when it insists on warmth and "a distinctly human feel" even inside a highly modular, token-driven system. Structure and warmth aren't in tension — a notebook is rigidly gridded *and* it's the thing you reach for because it feels like yours. That's the target.

## 1. Core Principles

These eight sentences settle every design argument you'll have with yourself later. If a component doesn't pass one of these, redesign it.

1. **Every element belongs to a unit.** Nothing floats at an arbitrary size or position. If it's on screen, it occupies whole grid cells or whole multiples of the base unit.
2. **Groups shrink together, not the page.** Responsiveness is renegotiated space *within* a row or column group, never a uniform scale-down of the whole layout.
3. **If it stays, it's not an overlay.** Anything meant to persist while you keep working (notes, a mini player, a status panel) is a docked widget or panel — not a floating card the user has to dismiss.
4. **Roundness is a signal, not a style.** Corner radius communicates *interactivity* and *containment* at two fixed sizes. It is never decorative, never large enough to fight the grid.
5. **Workflow beats decoration.** If a choice makes the interface prettier but slower to scan or operate, the workflow wins. Every single time.
6. **Compact, not cramped.** Default to the smallest size that still reads clearly at arm's length — not the smallest size that technically fits. Density comes from *tight internal spacing* (padding, line-height, icon-to-label gaps), never from *removed* spacing between unrelated groups. A dense layout still breathes between sections; it just refuses to waste space inside them. If you're ever tempted to make an element bigger "so it doesn't feel cramped," fix the spacing around it instead — the element is very likely already the right size.
7. **Things should respond, not just switch.** A state change (hover, press, open, close) should feel like something happened to a physical object, not like a value flipped in a database. This doesn't mean skeuomorphism — it means every interactive element gives a small, proportionate acknowledgment that it registered you. See §12.1.
8. **Spend your personality budget on a few moments, not every moment.** Standard controls (checkboxes, scrollbars, text selection, system menus) should behave exactly the way people already expect — don't reinvent them for the sake of looking distinctive. Save PaperDesign's actual character for the handful of moments that define the product: an empty state, a first-run screen, a widget nobody else's app has. A system that's "unusual" everywhere is exhausting; a system that's familiar everywhere except three well-chosen places is memorable.

> **On principle 6, specifically:** most first drafts of PaperDesign layouts run 15–25% too large across the board — hero type, buttons, badges, card padding. If a screen looks like a template from a website builder, it's almost always because every element was sized for "comfortable default" instead of "as small as clarity allows." When in doubt, size down one step from your instinct and see if anything actually got harder to read. It usually didn't.

> **On principles 7 and 8:** these are what keep a strict grid-and-token system (§2, §6) from reading as cold or mechanical. A layout can be perfectly compliant with every measurement in this spec and still feel lifeless if nothing ever *responds* to being touched, or if every single element is fighting to be noticed. Structure gives PaperDesign its clarity; restraint about *where* to spend warmth is what keeps that structure from feeling like a spreadsheet.

## 2. The Grid — the foundation of everything

PaperDesign is grid-first, the way a page of graph paper is grid-first. The grid is not a background pattern (although showing it, faintly, as a canvas texture is a legitimate and encouraged PaperDesign move — see §9) — it is the actual layout mechanism.

### 2.1 Base unit

- **Base unit = 8px.** All spacing, sizing, and positioning is expressed in multiples of 8 (`1u = 8px`). Half-steps of 4px (`0.5u`) are permitted only for hairline adjustments inside a component (icon-to-label gaps, border alignment) — never for outer spacing between components.
- Component *heights* snap to the 8px grid. Component *widths* are free within their column, since text and content are naturally variable-width — but their internal padding still snaps to 8px multiples.

### 2.2 Spacing scale

| Token | Value | Use |
|---|---|---|
| `space-0` | 0px | flush edges inside tightly grouped elements |
| `space-1` | 4px | icon-to-text gap, hairline internal padding |
| `space-2` | 8px | default internal component padding (1u) |
| `space-3` | 16px | space between related elements in a group (2u) |
| `space-4` | 24px | space between groups (3u) |
| `space-5` | 32px | space between sections (4u) |
| `space-6` | 48px | page-level margins on desktop (6u) |
| `space-8` | 64px | large canvas/section breaks (8u) |

Never use a value outside this scale. If nothing fits, that's a sign the layout is wrong, not that the scale needs an exception.

**One narrow exception, borrowed from Fluent's spacing ramp:** icon padding inside a small control (a 20px icon inside a 32px button) sometimes needs 2px or 6px to optically center rather than land exactly on an 8px multiple — icons have visual weight that isn't evenly distributed across their bounding box. This is the *only* place a non-scale value is allowed, and only for optical icon centering, never for general layout spacing.

### 2.3 Column grid

- Desktop canvas: **12-column grid**, gutters = `space-3` (16px), outer margin = `space-6` (48px), max content width 1440px before margins grow instead of columns stretching.
- Compact/mobile canvas: **4-column grid**, gutters = `space-2` (8px), outer margin = `space-3` (16px).
- Sidebars, panels, and widgets always span whole columns. A 3-column-wide sidebar next to a 9-column content pane is correct. A sidebar that's 287px because "it looked right" is not.

### 2.4 The notebook-grid canvas texture

For full-bleed backgrounds (desktop shells, lock screens, canvas apps), PaperDesign uses a **faint graph-ruled texture**: 1px lines at `space-4` (24px) intervals, at 4–6% opacity over the base surface color. This is the one place where the grid is *visible* rather than just structural — a quiet reminder of what's holding the layout together. It should never appear behind dense text content (only behind canvases, empty states, desktops, lock screens).

## 3. Responsive Behavior — "groups shrink, not pages"

This is PaperDesign's most opinionated stance, and it's the fix for the thing that makes most responsive design feel nauseating: the whole page scaling down as one blob, so spacing, icons, and text all shrink proportionally and unevenly.

### 3.1 The rule

A layout is a tree of **groups** — a row of toolbar icons, a column of nav items, a card grid. Each group has its own minimum content size and its own reflow behavior. When the viewport narrows:

1. Groups first give up their *internal spacing* (drop from `space-4` to `space-3` to `space-2`, in that order) before touching content size.
2. If a group runs out of spacing to give up, its members move to the next available reflow pattern for that group type (see 3.2).
3. **Global font size and icon size never scale down with viewport.** They scale only in discrete steps tied to explicit density modes (Comfortable / Compact — §7), which the user or context chooses, not the viewport.
4. Unrelated groups on the same screen do not need to hit the same breakpoint at the same time. A toolbar can compress before a content grid does.

### 3.2 Reflow patterns by group type

- **Icon row (toolbar/dock):** icons move closer together (spacing collapses `space-3 → space-2 → space-1`) before anything is hidden. Only after minimum spacing is hit does an overflow menu appear — and it appears as a docked group (a menu list), never a modal overlay.
- **Card grid:** column count drops (4 → 3 → 2 → 1) before card *content* shrinks. Cards keep their internal padding fixed; the grid gives up columns, not padding.
- **Text block / form:** never reflows internally — it just gets a narrower column. Font size is fixed regardless of width.
- **Nav list (sidebar):** collapses to icon-only (labels hide, icons remain at fixed size) before it collapses to a hamburger menu. Icon-only is a *state*, not a shrink.

### 3.4 Platform adaptivity — one token system, native shape per surface

Borrowed from Fluent 2's principle of reusing native platform patterns rather than forcing one visual output everywhere: PaperDesign's *tokens* (spacing, radius, type scale, color relationships) travel unchanged across a CircuitPython macropad display, a Flutter app, and a desktop shell — but the *components* built from those tokens should feel native to each surface rather than identically skinned.

- A macropad's tiny display doesn't get a scaled-down desktop layout — it gets its own component built from the same 8px unit and two-radius rule, designed for that size from the start.
- Don't port a component 1:1 across platforms just because the code is shared. Port the *tokens*; redesign the *layout* per surface.
- The one thing that must never vary by platform: the shape language (§5) and the "groups shrink, not pages" rule (§3.1). Those are what make it recognizably PaperDesign regardless of the screen it's running on.

### 3.5 What PaperDesign explicitly avoids

- Scaling an entire layout via a CSS `transform: scale()` or viewport-relative units (`vw` font sizes) for anything but the graph-paper texture itself.
- "Everything just gets 20% smaller" behavior. If a design needs that, the grid unit itself was wrong for that breakpoint — fix the unit, not the render.

## 4. Overlays vs. Widgets vs. Panels — a hard distinction

This is the single most-violated distinction in modern software, and PaperDesign treats it as load-bearing.

| | **Overlay** | **Widget** | **Panel** |
|---|---|---|---|
| Lifetime | Transient — dismissed by the user or by completing a task | Persistent — stays visible across the whole session | Persistent, but toggleable — user opens/closes it explicitly |
| Position | Floats above the layout, has its own elevation/shadow | Docked into the grid, occupies real column/row space | Docked, slides in/out from an edge, still grid-aligned when open |
| Backdrop | Dims or blurs what's behind it | None — it's just another grid citizen | Pushes or overlaps content, no dimming |
| Use for | Confirmations, quick pickers, context menus, toasts, onboarding tips | Notes, weather, now-playing, status, anything meant to be glanced at repeatedly | Settings, inspector/details, chat history, file browser sidebar |
| Never use for | Anything the user is expected to keep open while working | One-off transient confirmations | Anything that should interrupt the current task |

**The test:** if you would be annoyed that this thing disappeared when you clicked elsewhere, it was never an overlay — build it as a widget or panel. Image 3 (the lock-screen notes card) is doing this correctly already: notes are a widget docked into the layout, not a popup.

### 4.1 Chrome vs. content — a trust boundary

Borrowed from Firefox's Acorn system, which draws a hard line between "chrome" (the browser itself — trusted, system-controlled) and "content" (the page inside it — untrusted, dynamic, can't impersonate the browser). PaperDesign applies the same boundary anywhere the system renders content it doesn't fully control: a file preview, an embedded webview, a plugin panel, user-generated content in a card.

- **Chrome** (toolbars, nav, window frame, system dialogs) always uses `surface-canvas`/`surface-raised` and system typography — never lets rendered content bleed its own fonts, colors, or controls into that space.
- **Content** (previews, embeds, anything not authored by the system) is visually contained inside a bordered or insetted region — the user should always be able to tell, at a glance, "this part is the app, this part is just what the app is showing me."
- Practical rule: a fake button, fake menu, or fake system dialog can never be rendered *inside* a content region in a way that's visually indistinguishable from real chrome. If a preview needs interactive controls, those controls live in the chrome around the content, not layered on top of it.

This matters most in exactly the kind of shell/desktop work you're doing (Image 2) — a file manager or window manager is chrome; whatever's open inside a window is content, and PaperDesign should never let the two visually merge.

## 5. Shape Language

### 5.1 Corner radius — two sizes, period

PaperDesign uses exactly **two** radius tokens for the entire system. Anything larger reads as childish (this is the single biggest complaint about Material); anything with more than two sizes reads as inconsistent.

| Token | Value | Use |
|---|---|---|
| `radius-sm` | 4px | Containers: cards, panels, input fields, buttons, tooltips, dropdowns — "just enough to not be a sharp box, not enough to look soft" |
| `radius-pill` | 999px (fully rounded) | **Bars only**: search bars, tag/chip components, toggle tracks, progress pills — anything shaped like a horizontal capsule where the height defines the curve |

There is no `radius-lg`. There is no "big friendly rounded card." Cards are boxy (`radius-sm`). The only fully-round things are pill-shaped bars and true circular elements (avatars, single-glyph icon buttons, status dots) — never rectangular panels or big buttons.

### 5.1a A third shape, used narrowly: the beak

Borrowed from Fluent 2's shape vocabulary (rectangle, circle, pill, beak): a small triangular pointer that anchors a floating element to the control that spawned it — a tooltip pointing at the icon it describes, a context menu pointing at the row it was opened from. This is the **only** place a non-grid-aligned angle is allowed in the whole system, precisely because it's functional: it's the one shape whose entire job is to show *where something came from*. Beak width = 8px, height = 4px, same fill as the surface it extends from, no separate border. Never use a beak decoratively (e.g., as a design flourish on a card) — it exists solely to anchor transient overlays (§4) to their trigger.

### 5.2 Bar height rule

For any pill-shaped bar (search, input, tag), height is derived from the type inside it, not chosen arbitrarily:

```
bar-height = text-line-height × 1.15, rounded up to the nearest space-1 (4px)
```

At 14px body text (line-height ~19px), that's ~22px → round to 20px inner content height, plus `space-2` (8px) vertical padding top/bottom — giving a real bar height around **32–36px**, not 40+. That's the ceiling for a default search/filter bar; only touch-primary contexts (mobile, macropad-adjacent physical UI) push it to 40px for tap accuracy, and that's a density-mode decision (§13), not a default. **The radius is always `height / 2`** so the ends are true semicircles — this is what makes a pill read as intentional geometry rather than "rounded corners cranked to max."

### 5.3 Containers vs. bars — don't mix the language

Rectangular containers (cards, panels, dialogs, buttons that aren't pill-shaped) use `radius-sm` only. Never round a card's corners to match a pill's radius — a card that tries to look like a capsule looks like a mistake, not a style.

## 6. Color System

### 6.1 Philosophy

PaperDesign does **not** mandate a fixed palette, and it doesn't mandate light-over-dark, warm-over-cool, or any specific hue. Color — including whether the whole product is light, dark, or something in between — is entirely the developer's call, project by project. A blue-toned dark mode, a cream light mode, a slate-and-moss theme, a near-monochrome build: all of these are PaperDesign if they follow the *relationships* below. None of them are PaperDesign if they skip the relationships and just pick colors that look nice in isolation.

What PaperDesign actually mandates is the **method** — how your neutrals relate to each other, how your accent relates to its companions — not the values themselves. Two projects following this method can look nothing alike in hue and both still be unmistakably PaperDesign, the same way two Braun products in wildly different colorways both read as Braun. The relationships are the brand; the palette is just an instance of them.

The one true hard rule, regardless of theme: **paper is not pure white and ink is not pure black, in either direction.** A light theme's surfaces are never a stark `#FFFFFF`; a dark theme's surfaces are never a stark `#000000`. Both get a deliberate undertone (§6.2). Beyond that single rule, everything below is a set of relationships to hold, not values to copy.

### 6.2 Choosing your neutrals

Pick a single hue to tint every neutral in the system — warm cream, cool slate, blue-gray, deep charcoal-with-a-hue-cast for a dark-first product, whatever fits the project — and commit to that one undertone throughout, in both light and dark variants if you build both. Then define four steps along that hue:

| Token | Relationship | Role |
|---|---|---|
| `surface-canvas` | Your undertone, background-most step | Full-bleed background |
| `surface-raised` | Same undertone, lighter (light themes) or lighter-and-less-saturated (dark themes) than canvas | Cards, panels — sits "above" canvas |
| `surface-sunken` | Same undertone, darker than canvas (light themes) or darker still (dark themes) | Inputs, wells — "pressed into" the page |
| `ink-primary` / `ink-secondary` / `ink-tertiary` | Same undertone, three steps of contrast against your surfaces | Text and icon hierarchy — never pure black, never pure white |

Four steps, one undertone, done. If your neutrals need a fifth step to work, the first four weren't spaced correctly — go back to the ratios, don't add a token. (For a light theme, worked example: `surface-canvas` ~92–95% lightness / 8–15% saturation, `surface-raised` ~98–99% lightness / near-zero saturation, `surface-sunken` ~86–90% lightness, ink steps at ~12%/40%/65% lightness. For a dark theme, invert the lightness direction but keep the same undertone and the same four-step logic — canvas darkest, raised slightly lighter, sunken darker than canvas, ink light-on-dark in three contrast steps.)

### 6.3 Choosing your accent — and its dull companions

Pick **one** accent hue for primary actions and active states. This can be genuinely anything — dusty periwinkle, muted terracotta, deep moss, ink blue, a near-black accent on a pale theme, a bright cool blue on a dark theme. The requirement isn't the hue, it's the **saturation ceiling**: keep it in the muted/dusty range (roughly 35–55% saturation at a mid-to-low lightness), never a vivid, poster-saturated color straight off a default web-builder swatch. If it looks like it came out of a crayon box, pull the saturation down until it looks like it came out of a paint chip instead.

For **secondary/tag/label colors** (category chips, semantic states, data-viz series) — the mistake almost every "generic new-gen" template makes is pairing one saturated accent with a rainbow of equally saturated companions (bright green, bright orange, bright purple tags sitting in a row). PaperDesign's rule: **companions are duller than the primary accent, not different in kind.** This is the one place PaperDesign is genuinely prescriptive about color, because it's a relationship, not a hue choice — the developer still picks *which* hues the companions use, PaperDesign just sets the ceiling they can't cross.

- Derive companions by shifting hue while *holding saturation and lightness at or below* your primary accent's values — never above.
- If your primary accent sits at 45% saturation, your tag/label palette should sit around 25–35% saturation. Muted sits next to muted; nothing in a supporting role should out-shout the one color that means "primary action."
- Semantic colors (success/warning/danger) follow the same ceiling — a muted sage, a muted ochre, a muted brick, not a stoplight.

### 6.4 Rules

- One accent color per screen for *primary actions*. If two things compete for accent color, one of them is wrong, not the palette.
- Never use pure `#000000` or pure `#FFFFFF` anywhere in the system, in either a light or dark theme.
- Dark mode is not "invert the palette" — it's a parallel *dark* scale in the same undertone, low-lightness surfaces rather than inverted ones, so the product doesn't feel like a different product at night. (And to be clear: a project can be dark-*first*, with no light mode at all, or the reverse — PaperDesign has no default direction here.)
- If you want a concrete starting point rather than deriving your own: undertone = warm cream (~40° hue), accent = dusty periwinkle (`~225° hue, 40% sat, 65% light`). That's one example, not a requirement — swap it, or the whole undertone family, freely per project. A cool-blue dark theme with a warm-orange accent follows this system exactly as correctly as the cream-and-periwinkle default does, as long as the companion-color ceiling in §6.3 holds.

## 7. Typography

### 7.1 Typeface

Typeface, like color, is the developer's choice — PaperDesign fixes the *scale and rhythm* (§7.2), not the letterforms. That said, here's the guidance worth following:

- **Too much uniformity ruins things.** A single font at a single weight everywhere is safe but flat — it's part of why a lot of "clean minimal" UI ends up feeling characterless rather than calm. A little contrast, deliberately placed, is what makes a system feel considered instead of default.
- **The recommended move: pair two typefaces that contrast while staying cohesive** — not two fonts that fight. A classic version of this is a serif for headlines/titles paired with a humanist sans for body/UI text (think Apple's own editorial pairings, or a New York-style serif over a San Francisco-style sans) — the serif gives titles some warmth and presence, the sans keeps body copy and controls efficient and legible at small sizes. A monospace third voice is a legitimate addition specifically for code, data, or timestamps (as in your Flux docs reference) — never for general body text.
- **What makes a pairing "cohesive" rather than "fighting":** similar x-height and stroke contrast, so the two faces sit at the same visual weight even though their letterforms differ. If one face looks noticeably heavier or more decorative than the other at matching sizes, they're fighting, not pairing.
- **If you'd rather not pair:** a single humanist sans-serif family at two weights (Regular 400, Semibold 600) is still a fully valid PaperDesign default — think Apple's San Francisco, Inter, or Public Sans register rather than a cold geometric/grotesque face like Roboto, which reads as "corporate SaaS" (the Material-adjacent look). This is the safe baseline; the serif/sans pairing above is the recommended upgrade once you want more character.
- Whatever you choose, hold it to two weights per face, and don't introduce a third typeface beyond an optional headline serif, an optional monospace, and your body sans. Three total voices is the ceiling — past that, it stops reading as "considered contrast" and starts reading as "no one decided."

### 7.2 Scale

A modular scale at **1.2× (minor third)**, base **14px** — one full step down from what almost every "new-gen SaaS" template ships with (16px base, 1.25–1.333 ratio, 40–56px hero type). PaperDesign's headlines are restrained on purpose: a notebook doesn't shout its section titles, it just labels them clearly and moves on.

| Token | Size | Weight | Use |
|---|---|---|---|
| `type-display` | 34px | 600 | Clock/time displays, hero numerals — used almost nowhere else |
| `type-h1` | 24px | 600 | Page titles, marketing headlines — this is the **ceiling**, including for hero copy |
| `type-h2` | 20px | 600 | Section headers |
| `type-h3` | 17px | 600 | Card/group titles |
| `type-body` | 14px | 400 | Default body text |
| `type-body-sm` | 13px | 400 | Secondary text, metadata |
| `type-caption` | 11px | 400 | Timestamps, hints, fine print, badge/pill labels |

Line height: 1.35× for body sizes, 1.1× for display/headline sizes — tight, not airy. Letter-spacing stays at 0 except `type-caption`, which gets +1.5% for legibility at small size.

If you've paired a headline serif with a body sans per §7.1, the serif takes `type-display` through `type-h3`; the sans takes everything `type-body` and smaller, plus every UI control regardless of size (buttons, inputs, nav — controls stay in the sans even inside an otherwise serif-headed layout, since serif at small interactive sizes tends to lose crispness). The size tokens don't change based on which face is assigned to them — the scale is fixed, the letterforms riding on it are not.

**On hero headlines specifically:** `type-h1` (24px) is the largest text allowed anywhere outside a literal clock/timer display. A landing-page headline set at 40–56px bold is a Material/generic-SaaS reflex, not a PaperDesign one — if it needs to command attention, use restraint and whitespace around a 24px line, not size. This is the single most common way a PaperDesign draft slides back into looking like a template.

### 7.3 Rule of restraint

No more than two type sizes visible in a single component. A card gets a title (`h3`) and a body (`body` or `body-sm`) — not four competing sizes fighting for attention.

## 8. Elevation & Depth

PaperDesign has three depth levels, matching the Overlay/Widget/Panel distinction in §4 — depth communicates *category*, not decoration.

| Level | Shadow | Use |
|---|---|---|
| `elevation-0` (flat) | none | Canvas, docked widgets, panels — anything grid-native |
| `elevation-1` (raised) | `0 1px 2px rgba(28,26,22,0.08), 0 1px 1px rgba(28,26,22,0.04)` | Cards at rest, dropdowns |
| `elevation-2` (floating) | `0 8px 24px rgba(28,26,22,0.14)` | True overlays only — dialogs, popovers, toasts |

No level 3+. If something needs more shadow than `elevation-2` to feel "above" the page, it's competing with something else that shouldn't be there.

## 9. Texture & Materiality — where "paper" actually shows up

This is the part that differentiates PaperDesign from a generic flat/neutral system:

- **Graph-paper canvas** (§2.4) on shell/desktop/lock-screen backgrounds — faint, structural, never decorative-only.
- **No gradients as decoration.** Gradients are permitted only functionally (a scrim behind text on a photo background, as in the lock-screen reference) — never as a stylistic flourish on buttons or cards.
- **No skeuomorphic paper texture** (no fake fibers, no drop-shadows pretending to be folded corners). The "paper" is a geometry and structure metaphor, not a literal texture simulation — that would tip into kitsch.
- **Hairline borders over shadows where possible.** A 1px `ink-tertiary` border at low opacity often communicates "this is a separate surface" more honestly than a shadow does — reserve shadow for true elevation (§8).

### 9.1 Illustration & imagery

Anthropic's own brand work leans on a warm, human illustration language rather than stock photography or generic 3D-render clip art (the free-icon-pack look every AI-generated landing page defaults to) — that's worth borrowing directly for empty states, onboarding, and error screens:

- Prefer simple line illustration in the same stroke weight as your iconography (§10) over photographic or 3D imagery. It keeps the whole system in one visual register instead of switching materials between "flat UI" and "rendered photo" mid-screen.
- Illustration is one of the few places outside the accent color where a second, warmer hue is allowed to appear — but it should still sit inside the muted/dusty saturation ceiling from §6.3, not introduce a vivid palette the rest of the system doesn't use.
- Reserve illustration for principle 8's "few moments that matter" (§1) — an empty state, a first-run screen, a 404. Don't illustrate routine UI (a settings page doesn't need a mascot); that's decoration competing with workflow, which principle 5 rules out.

## 10. Iconography

- Stroke-based, not filled, at a consistent **1.5px stroke weight**, 20×20 or 24×24 canvas depending on context (toolbar = 20, empty-state/illustration = 24+).
- Corners on icon geometry follow the same two-radius logic as §5 — small joints, not bubble-rounded.
- Icons are **never** the only affordance for a destructive or primary action — always paired with a text label unless space is provably exhausted (icon-only toolbar after all spacing options in §3.2 are used).

## 11. Components — behavior specs

### 11.1 Buttons
- Rectangular, `radius-sm`, height **32px by default** (`space-4`), 40px only in Comfortable density or touch-primary contexts (§13) — never 40px as the baseline.
- Horizontal padding `space-3` (16px), never more — a button's width is set by its label plus that padding, not by matching a neighboring element's width.
- Type is always `type-body` or `type-body-sm`, 600 weight. Never `type-h3` or larger inside a button, even a hero CTA — the button gets its emphasis from being the accent-filled element on the screen, not from oversized text.
- Primary: `accent-primary` fill, `ink-on-accent` text.
- Secondary: `surface-sunken` fill, `ink-primary` text, 1px `ink-tertiary` border.
- Ghost/tertiary: no fill, no border, `accent-primary` text — for the least important action in a group.
- Only pill-radius buttons: none. Buttons are boxy per §5.1.

### 11.2 Inputs / text fields
- `surface-sunken` fill, `radius-sm`, 1px border in `ink-tertiary` at rest, `accent-primary` on focus (2px focus ring, no glow/blur).
- Default height **32–36px** per the recalculated bar rule in §5.2 — not 40px+.
- Search/filter fields specifically: pill-radius per §5.2, since they function as bars, not boxes — this is the one exception where `radius-pill` applies to an input.
- Icons inside inputs sit at `space-2` from the edge, vertically centered, `ink-secondary` color, sized 16px (not 20–24px — an oversized search icon in a compact bar is a common tell that sizing wasn't checked against the type scale).

### 11.3 Cards
- `surface-raised`, `radius-sm`, `elevation-1`, internal padding `space-3` or `space-4` depending on density mode.
- Cards never nest cards. A card containing another card is a sign the inner content should be a list row or a section divider instead.

### 11.4 Navigation (sidebar / rail)
- Column-grid aligned, `elevation-0`, fixed icon size regardless of collapse state.
- Reflow per §3.2: labels hide before icons shrink; icons never shrink.
- Active state = `accent-primary-muted` background fill on the row, not a colored icon alone (color-only state changes fail accessibility and are easy to miss).

### 11.5 Dialogs / modals (true overlays)
- `surface-raised`, `radius-sm`, `elevation-2`, `surface-overlay-scrim` backdrop.
- Reserved strictly for things requiring a decision before the user can continue. If dismissible without consequence, it should probably be a toast or inline message instead, not a modal.

### 11.6 Toasts / inline confirmations
- `elevation-2`, docked to a fixed screen corner, auto-dismiss, never blocks input to the rest of the page (no scrim).

### 11.7 Widgets (docked, persistent)
- `elevation-0` or `elevation-1` at most (they belong to the page, not above it).
- Fixed grid span (whole columns/rows). Internal content follows §3.2 reflow rules independently of the rest of the page.

## 12. Motion

### 12.1 Duration scales with what's moving, not a fixed number for everything

Borrowed from Fluent 2's motion guidance: bigger elements and longer travel distances need more time to read as natural, or they look like they teleported. PaperDesign doesn't use one blanket duration — it ties duration to what's actually moving:

| Scale | Duration | Use |
|---|---|---|
| Micro | **100ms** | Hover/press feedback, checkbox toggle, focus ring — small, local changes |
| Default | **180ms** | Dropdown open/close, tab switch, small panel reveal |
| Macro | **260ms** | Full panel slide, dialog entrance, page-level transition — large elements, longer travel |

- Easing: `cubic-bezier(0.2, 0, 0, 1)` (confident decelerate) for anything entering; `cubic-bezier(0.4, 0, 1, 1)` (accelerate) for anything exiting. No bounce, no elastic overshoot.
- Panels and widgets slide along the grid axis they're docked to — one motion property change at a time (position *or* opacity, rarely both at once) keeps it feeling mechanical and precise, "paper sliding," not "UI bouncing."
- **Always respect reduced-motion settings.** If the OS/browser requests reduced motion, collapse everything to an opacity crossfade under 100ms, no movement. This isn't optional polish — treat it the same as any other accessibility requirement.

### 12.2 Light & feedback — the paper-metaphor version of Fluent's "Reveal"

Fluent's Light principle uses a soft glow under the cursor to signal interactivity — that's a glass/acrylic-material idea, and PaperDesign's material is paper, not glass, so it's reinterpreted rather than copied directly. The underlying goal carries over exactly: **an interactive element should visibly acknowledge the cursor/touch before it's even clicked** — this is what makes principle 7 (§1) real instead of aspirational.

- **Hover:** surface shifts one step warmer/darker on the existing neutral scale (`surface-raised → surface-sunken`-adjacent, roughly a 4–6% lightness shift) — like pressing a pencil slightly harder, not like a light turning on. No glow, no blur, no drop-shadow growth.
- **Press/active:** an additional 4–6% shift plus a 1px inset — the element visibly "gives" under the click, then returns on release. Duration: Micro (100ms) per §12.1.
- **Focus (keyboard):** a 2px `accent-primary` ring, offset 2px from the element — never rely on the hover shift alone for focus, since it isn't visible to keyboard-only or screen-reader users.
- This feedback is required on every clickable element without exception — a button, row, or icon with zero visual response to hover is the single fastest way an interface starts to feel mechanical rather than natural, regardless of how correct its grid and spacing are.

## 13. Density & Scale Modes

Three explicit modes, user- or context-selected — this is how PaperDesign supports "compact" without ever silently shrinking things via viewport math (§3.1, rule 3), and how it supports going bigger and friendlier without silently drifting into Material's bloat.

| | Compact (default) | Comfortable | Expressive |
|---|---|---|---|
| Row height | 32px | 40px | 48px |
| Button/bar height | 32px | 40px | 48px |
| Card padding | `space-2` (8px) | `space-3` (16px) | `space-4` (24px) |
| Type scale | as §7.2 | unchanged (never shrink text for density — only spacing) | base shifts to 16px, same 1.2 ratio, `type-h1` ceiling shifts to 28px |

**Compact is the PaperDesign default**, not an opt-in power-user mode — this is the "compaction without crampedness" principle (§1.6) made literal. Comfortable and Expressive exist for touch-primary contexts, accessibility needs, or simply a developer's deliberate choice to make a product feel bigger and friendlier — that's a legitimate direction, not a lapse, as long as it's chosen on purpose rather than crept into by instinct-sizing (the exact failure mode §1.6's callout warns about).

### 13.1 Going bigger without losing the Paper aesthetic

Making a PaperDesign product bigger and friendlier is fully supported — the trap is *how* most systems get bigger: they round the corners more too, and that's what turns "modern and friendly" into "generic and childish." Scale size up; keep the following exactly fixed regardless of mode:

- **Radius never scales with size.** `radius-sm` is 4px in Compact and 4px in Expressive. A bigger button does not get a bigger corner radius — this single rule is what keeps an Expressive-mode PaperDesign screen from turning into Material. If it needs to look softer at a larger size, that's a job for more internal padding, not a bigger radius.
- **The two-shape rule (§5) never gains a third size.** Boxes stay `radius-sm`; bars stay pill. Expressive mode just means bigger boxes and longer bars, not new shape categories.
- **Type ratio stays fixed; only the base shifts.** Expressive moves the base from 14px to 16px and keeps the 1.2 scale ratio — it doesn't switch to a steeper ratio just because things are bigger. Headlines still don't shout; they just sit on a slightly larger grid.
- **The grid unit itself can move (8px → 10px or 12px) for a genuinely larger product** (e.g., a touch-first tablet app, physical hardware with bigger targets like a macropad faceplate) — but it has to move as a deliberate, documented decision for that project, applied consistently, not crept into per-component. Pick one base unit for the whole product and hold it, the same discipline as §2.1 at whatever scale you land on.
- **More space goes between groups, not inside them, as you scale up.** This is principle 6 (§1) again at a bigger size: an Expressive layout should still feel like it's using its size to be *legible from further away*, not to be *emptier*. If Expressive mode just means "the same layout with 1.5× padding everywhere," that's inflation, not friendliness — spend the extra room on breathing room between sections and on making touch targets genuinely bigger, not on padding a button that already reads fine.

If a screen passes all five of the above at Expressive scale, it's still recognizably PaperDesign — bigger, warmer, friendlier, and still built on a grid with two shapes and one accent, which is the whole point.

## 14. Voice & Content

- Sentence case everywhere (buttons, headers, nav labels) — not Title Case, not ALL CAPS. Title Case reads as corporate-brochure; PaperDesign is a tool, not a brochure.
- Labels describe the action, not the icon: "Log in," not "Submit." "Delete file," not just "Delete" if ambiguity is possible.
- Errors state the problem and the fix in one short sentence: *"Password must be at least 8 characters."* — not *"Invalid input."*
- No exclamation points in system copy except genuine celebration states (onboarding complete, task done) — and even then, sparingly. (Small note on your reference: "Notes!" is charming for a personal build, but the spec default is calm, unexclaimed labels — "Notes" — to keep the system tone consistent across contexts you didn't personally hand-letter.)
- **A little restrained warmth is allowed, deliberately.** Firefox's Acorn principles put it well: engaging but never in the way. PaperDesign is not required to be sterile — an empty state can have a dry, quiet bit of personality ("Nothing logged today. A blank page is still a page."), a loading state can have one understated wry line instead of a generic spinner label. The test is the same as everywhere else in this spec: does it help or slow down the scan? One charming line in an empty state helps. Charming copy on every button does not.

## 15. Anti-Patterns — what PaperDesign explicitly is not

Keep this list taped above the monitor. If a design decision matches one of these, stop.

- ❌ Corner radius above 4px on anything that isn't a pill-shaped bar or a true circle.
- ❌ A floating card/panel that the user is expected to keep open while doing other work (→ make it a widget).
- ❌ The whole page scaling down together on resize instead of groups renegotiating their own spacing.
- ❌ More than one accent color competing for attention on a single screen.
- ❌ Icon-only controls for anything destructive or non-obvious.
- ❌ Drop shadows on flat, grid-native elements (widgets, panels, canvas content) — shadows are reserved for true overlays and resting cards only.
- ❌ Decorative gradients on buttons/cards "for depth."
- ❌ Bounce/elastic motion easing.
- ❌ Mixing pill-radius and box-radius on the same visual family of component.
- ❌ Text or icons that shrink because the *window* got smaller rather than because the *density mode* changed.
- ❌ Hero/headline type set above 24px (`type-h1`). This is the fastest way a screen slides back into generic-template territory.
- ❌ Default button or input height above 32–36px "because it looked more comfortable" — that instinct is what §1.6 exists to override.
- ❌ A pill-shaped "eyebrow" badge announcing a version number or feature above a hero headline. It's a template reflex, not a functional element — if the info matters, put it in a changelog widget (§4), not a floating badge nobody reads twice.
- ❌ Decorative fake window chrome (traffic-light dots) on a screenshot/demo frame unless the artifact being shown is *actually* a desktop window in-product — using it purely as a framing device around a marketing screenshot is set-dressing, not function, and it's one of the clearest generic-SaaS tells.
- ❌ A rainbow of equally-saturated tag/label colors. Companions to the accent must be duller than the accent (§6.3), never competing with it.
- ❌ Designing in Comfortable density by default instead of Compact (§13).
- ❌ Using the beak/pointer shape decoratively, or on anything angled that isn't literally anchoring a transient overlay to its trigger (§5.1a).
- ❌ Rendered content (previews, embeds, third-party panels) styled closely enough to system chrome that a user can't tell which one they're looking at (§4.1).
- ❌ A clickable element with zero hover/press feedback — no visual acknowledgment that it registered the cursor (§12.2).
- ❌ Reinventing a standard control (checkbox, scrollbar, system menu) for the sake of looking distinctive, instead of spending that distinctiveness on the few moments that actually define the product (§1, principle 8).
- ❌ Generic stock photography or 3D-render clip art in empty states — use line illustration in the system's own stroke weight instead (§9.1).
- ❌ Increasing corner radius as a component gets bigger in Comfortable/Expressive mode. Radius is fixed at every scale (§13.1) — size up with padding and grid unit, never with roundness.
- ❌ Assuming PaperDesign requires a specific hue, or requires light-over-dark. Both are the developer's choice (§6.1) — the only fixed things are the *relationships* (four neutral steps, one undertone, companions duller than accent).
- ❌ Pairing two typefaces that don't share a comparable x-height/weight, so one visibly dominates the other instead of contrasting cohesively (§7.1).
- ❌ More than three total type voices in one product (headline face + body face + optional monospace is the ceiling).

## 16. Quick Reference Card

```
UNIT           8px default (may shift to 10/12px per-project, held consistently)
RADIUS         4px (containers) · pill (bars & circles only) · beak (tooltip/menu anchors only)
               — FIXED at every density/scale mode. Never grows with size.
TYPE SCALE     14/16/(your call) base × 1.2 ratio · h1 ceiling scales w/ mode, never past ~28px
TYPEFACE       developer's choice · recommended: contrasting-but-cohesive pairing
               (headline serif + body/UI sans), 3-voice ceiling incl. optional mono
BUTTON/BAR     32 (Compact, default) / 40 (Comfortable) / 48 (Expressive)
DENSITY        3 modes: Compact (default) · Comfortable · Expressive — chosen, never crept into
COLOR          fully developer's choice — hue, light/dark, everything;
               PaperDesign fixes the METHOD only: 1 undertone/4 steps for neutrals,
               1 muted accent, companions duller than accent, never pure black/white
ACCENT         1 per screen, max
SHADOW         0 = flat/grid-native · 1 = resting card · 2 = true overlay only
MOTION         100 / 180 / 260ms, scaled to element size · decel-in / accel-out · no bounce
FEEDBACK       every clickable element visibly responds to hover/press — no dead controls
RESPONSIVE     groups shrink spacing → reflow pattern; page never scales as one
PLATFORM       tokens travel everywhere; components redesign natively per surface
CHROME/CONTENT system UI and rendered content are always visually distinguishable
PERSONALITY    familiar patterns everywhere, distinctive character in a few chosen moments
PERSISTENT UI  → widget/panel, docked, grid-aligned. never a floating overlay.
```
